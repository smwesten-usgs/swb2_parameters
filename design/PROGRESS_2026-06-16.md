# PROGRESS — 2026-06-16

## Summary

Consolidated the codebase from two divergent versions (v1 in `src/`, v2 in `v2/`) into a single clean implementation at `src/swb2_parameters/`. The new code is TOML-driven, uses subcommands, and has a simplified long-form schema with a single `lu_code` column.

---

## Key Design Changes (from v1/v2)

### 1. Single `lu_code` column replaces `lu_cdl` + `lu_nlcd`

The old schema had both `lu_cdl` and `lu_nlcd` columns with a `primary_key` setting selecting which one became `lu_code` at output time. This was dropped in favor of:

- Long tables have **one** `lu_code` column — it contains whichever coding scheme (CDL, NLCD, or custom) the project uses.
- A **crosswalk table** (`example/cdl_nlcd_crosswalk.tsv`) documents the CDL↔NLCD mapping as pure reference — it's not read by the tool.
- If you need an NLCD-keyed model, you author a separate long-form file with NLCD codes as `lu_code`.

**Rationale:** The many-to-one CDL→NLCD mapping (103 CDL codes → NLCD 82) makes automatic aggregation scientifically unsound. Better to make that a deliberate authoring decision.

### 2. TOML drives everything

No more CLI flags for families, singletons, primary_key, etc. The project TOML defines:

```toml
[long_schema]
hsg_count = 7

[groups.smgrain]
codes = [21, 22, 23, ...]

[tables.lu_lookup]
families = ["cn", "rz"]
singletons = ["max_net_infil_1", ..., "growing_season_interception_a", ...]

[tables.irr_lookup]
families = []
singletons = ["kcb_ini", "kcb_mid", ...]
```

Each `[tables.<name>]` section produces one output `.tsv`.

### 3. Subcommand CLI

```bash
swb2-parameters build <inputs>... --config project.toml --outdir outputs/
swb2-parameters reverse <inputs>... --config project.toml --outdir baseline/
swb2-parameters pest-export --config project.toml --outdir pest_files/
```

### 4. `max_net_infil` as singletons (not a family)

The aligner-based expansion (A-soil × fixed factors) was the weakest of the three families. In practice, max net infiltration is a soil property that doesn't vary by land use. It's now assigned via an ALL template (`max_net_infiltration.tsv`) with explicit per-HSG values as singletons.

### 5. Separate files for template types

The input design encourages splitting parameters by authoring concern:

| File | Purpose |
|------|---------|
| `lu_params_long_all_cdl.tsv` | Code-specific LU params (per crop/landuse) |
| `irr_params_long_all_cdl.tsv` | Code-specific IRR params (per crop/landuse) |
| `lu_params_long_group_smgrain.tsv` | Named group template (all small grains share these) |
| `irr_params_long_group_smgrain.tsv` | Named group template (IRR for small grains) |
| `rew_and_tew.tsv` | ALL template (soil-driven, same for every code) |
| `max_net_infiltration.tsv` | ALL template (soil-driven, same for every code) |

All files are passed as positional args to `build`. The tool concatenates them and applies group precedence: ALL < named group < code-specific.

## Current Long-Form Schema

```
lu_code, description, group, column, parlbnd, parubnd, parval1,
units, notes, ref, drained_condition, fixed_parval1, num_decimals
```

- **Template rows:** `lu_code` blank, `group` non-empty (e.g., "ALL", "smgrain")
- **Code-specific rows:** `lu_code` populated, `group` blank
- **FIXED mode:** `fixed_parval1` non-empty → value passed through, no bounds/rounding
- **BOUNDED mode:** `parlbnd`/`parubnd` required; `parval1` rounded per family defaults or `num_decimals`

## What's Not Yet Implemented

### `reverse` subcommand
Wide → long baseline. The v1 implementation (`reverse.py`) had working logic:
- Detect `lu_code` column
- Detect `description*` columns
- Emit one long row per parameter column
- Leave metadata fields blank (bounds, refs, etc.)

Needs to be reimplemented against the new simplified schema.

### `pest-export` subcommand
PEST++ artifact generation. Design doc exists at `design/PEST_EXPORT_DESIGN.md`. Core tasks:
1. Generate `.tpl` files from wide tables (fixed columns as literals, parameterized as `~ parnme ~`)
2. Generate `par_data.csv` (parnme, partrans, parval1, parlbnd, parubnd, pargp)
3. Generate `pargp_data.csv`
4. Parameter naming: `{table}_{param}:{lu_code}` (e.g., `lu_cn_2:1`)
5. Bounds sourcing: prefer long-form bounds, fallback to ±margin
6. Zero-value handling: `parval1 = 0` → `partrans = fixed`

### Additional group templates
Only `smgrain` and `smveg` are defined. Other potential groups:
- Row crops (corn, soybeans, sorghum)
- Tree crops / orchards
- Developed lands (CDL 121–124)
- Forests (CDL 141–143)

### Validation enhancements
- Warn when a TOML singleton is missing from the materialized data
- CN monotonicity check (CN_A ≤ CN_B ≤ CN_C ≤ CN_D)
- Optional strict mode (fail instead of warn)

### Documentation
- Update `README.md` to reflect new CLI/schema
- Update `design/README.md` and `design/NOTES.md` to mark v1 concepts as superseded
